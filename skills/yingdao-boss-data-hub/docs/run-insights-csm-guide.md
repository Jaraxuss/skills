# 客户成功行动看板：运行洞察说明

## 1. 为什么做运行洞察

BOSS 现有图表已经能看到客户的运行统计：

- 数据报表：运行次数、运行时长、账号活跃数、应用活跃数、健康度等整体趋势。
- 应用列表：每个应用的 owner、上周/月运行次数、运行小时、总运行小时、状态等。
- 应用详情：单个应用在时间维度上的运行次数和运行时长趋势。

这些信息能回答：客户有没有在运行、哪些应用跑得多、某个应用最近趋势如何。

但 CSM 做客户经营判断时，还需要回答更具体的问题：

- 每天活跃账号背后，是不是同一批真实业务账号？
- 活跃数上涨，是客户内部扩散，还是少数账号跑得更频繁？
- 应用 owner 是否等于真实运行者？
- 哪些应用已经进入客户业务流程，可以作为续费或复盘价值证据？
- 哪些高价值应用只依赖 1-2 个账号，存在单点风险？
- 下一次客户访谈应该优先找哪些真实使用者？

运行洞察不是替代 BOSS 的统计图表，而是在 BOSS 统计之上补齐“人、应用、关系、持续性”这层客户成功视角。

一句话定位：

> BOSS 现有图表告诉我们“客户有没有在用”；运行洞察进一步告诉我们“谁在用、用哪些核心应用、是不是持续在用、有没有形成业务价值”。

## 2. 功能入口

在客户成功行动看板中，打开客户详情，切换到 `运行洞察` 页签。

运行洞察包含三个区块：

1. 核心应用概览
2. 运行账号 x 日期活跃图
3. 核心应用 x 运行者矩阵

如果当前客户没有接入运行明细，会展示空态：

```text
当前客户未接入运行明细，无法展示运行洞察。
```

## 3. 核心应用概览

### 解决什么问题

帮助 CSM 从客户的应用运行数据中，识别并沉淀“核心应用资产”。

BOSS 应用列表能展示所有应用及其运行统计，但不区分哪些应用对客户成功最关键。运行洞察只保留一个人工标签：`核心应用`。CSM 可以结合运行次数、运行时长、活跃天数、运行者数和业务理解，把真正重要的应用标出来。

### 怎么看

表格字段包括：

| 字段 | 含义 |
|---|---|
| 核心 | CSM 手动标记的核心应用星标 |
| 应用 | 应用名称，名称缺失时展示应用 uuid 后 8 位 |
| 运行次数 | 当前时间范围内的运行开始次数 |
| 运行时长 | 当前时间范围内累计运行时长 |
| 活跃天数 | 当前时间范围内有运行的天数 |
| 运行者数 | 当前时间范围内真实运行过该应用的账号数 |
| 最近运行 | 当前时间范围内最近一次运行时间 |

### 业务价值

- 把运行数据沉淀成客户核心应用资产。
- 续费、复盘、客户访谈时，可以优先围绕核心应用讨论价值。
- 降低应用噪音，避免 CSM 在大量低价值或测试应用中反复筛选。

### 使用建议

首次查看客户时，建议按运行次数和运行时长排序，先标记 5-10 个核心应用。核心应用标记保存在当前浏览器，下次打开同一客户仍会保留。

## 4. 运行账号 x 日期活跃图

### 解决什么问题

回答：客户是不是有一批真实账号在持续使用。

BOSS 里可以看到每天账号活跃数，但看不到每天活跃账号是谁，也看不到是不是同一批人在持续使用。这个热力图把“账号活跃数”拆成具体账号和日期。

### 怎么看

- 行：真实运行账号。
- 列：日期。
- 单元格：当天运行次数。
- 颜色越深：当天运行越频繁。
- 空白：当天没有运行。

顶部摘要帮助快速判断整体情况：

| 指标 | 含义 |
|---|---|
| 真实运行账号 | 当前时间范围内至少运行过一次的账号数 |
| 日均活跃账号 | 平均每天有多少账号运行 |
| 连续活跃账号 | 活跃天数达到时间范围 60% 以上的账号数 |
| 新增 / 沉默 | 本周期新增活跃、或从上周期活跃转为本周期不活跃的账号 |
| Top3 占比 | 运行次数最高的 3 个账号占总运行次数比例 |

### 典型判断

- 如果每天都有一批账号连续出现，说明使用稳定。
- 如果活跃账号很多，但大部分只偶发出现，说明扩散还不稳定。
- 如果 Top3 占比很高，说明使用可能集中在少数账号。
- 如果近期出现一批新增账号，说明客户内部推广可能有效。
- 如果过去活跃账号近期大片空白，说明存在沉默或业务中断风险。

### 业务场景

- 续费前判断客户是否真实持续使用。
- 复盘时识别关键使用者和沉默账号。
- 客户声称“大家都在用”时，用真实账号分布验证。
- 活跃数上涨时，判断是真扩散，还是少数账号运行频次增加。

一句话：

> 它把“每天多少账号活跃”变成“哪些账号在持续活跃”。

## 5. 核心应用 x 运行者矩阵

### 解决什么问题

回答：核心应用到底被哪些真实账号运行。

BOSS 应用列表能看到应用 owner 和应用运行统计，但 owner 不等于真实运行者。应用详情能看到单个应用趋势，但看不到这个应用是一个人在跑，还是团队多人在跑。矩阵把“应用”和“真实运行账号”放在同一张图里。

### 怎么看

- 行：应用。
- 列：真实运行账号。
- 单元格：该账号在当前时间范围内运行该应用的次数或时长。
- 颜色越深：运行越集中或频繁。
- 空白：该账号没有运行该应用。

点击有数据的单元格，可以查看：

- 运行次数
- 运行时长
- 活跃天数
- 首次运行
- 最近运行

### 典型判断

- 横向看一个应用：判断它是团队共用，还是只被少数账号运行。
- 纵向看一个账号：判断这个账号是不是多个核心应用的关键使用者。
- 核心应用覆盖账号越多，越能说明应用已经进入客户业务流程。
- 高运行应用如果只集中在 1 个账号，需要关注单点依赖或账号迁移风险。

### 业务场景

- 续费前整理价值证据：哪些核心应用被哪些真实账号持续运行。
- 客户访谈前找对象：优先找运行核心应用最多、最稳定的账号。
- 识别单点风险：高价值应用只依赖少数运行账号。
- 判断应用是否从“开发完成”进入“业务使用”。

一句话：

> 它把“应用跑了多少次”变成“核心应用被哪些人实际使用”。

## 6. 推荐使用流程

1. 先看 `运行账号 x 日期活跃图`，判断客户整体使用是否稳定、是否扩散、是否集中。
2. 再看 `核心应用概览`，把真正有业务价值的应用标记为核心应用。
3. 然后看 `核心应用 x 运行者矩阵`，确认核心应用被哪些真实账号使用。
4. 最后形成 CSM 行动：
   - 续费价值证据
   - 客户访谈对象
   - 沉默账号跟进
   - 核心应用扩散建议
   - 单点依赖风险提醒

## 7. 注意事项

- 运行洞察基于详细 Excel 运行明细聚合，不直接展示全部明细。
- 需要逐条证据时，仍然回到 Excel 明细核查。
- V1 不使用运行状态、启动方式、运行方式做判断。
- 核心应用标记只保存在当前浏览器，不做团队共享。
- 选择 `核心应用` 范围时，如果没有标记核心应用，热力图会展示空态；可以先标记核心应用，或切换到 `全部应用` 查看。
