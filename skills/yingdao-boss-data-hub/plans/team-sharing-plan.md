# 客户成功行动看板 · 团队分享规划

> 面向：CS/客户成功团队
> 时长建议：30-40 分钟（含演示 + 讨论）
> 目标：让团队理解看板解决什么问题、怎么用、怎么拿到手

---

## 一、分享结构总览

| 环节 | 时长 | 内容 |
|------|------|------|
| 1. 为什么做这个看板 | 5 min | 痛点 → 目标 |

## 二、各环节详细内容

### 环节 1：为什么做这个看板（5 min）

**开场用数据制造共识**（基于 6 月 19 日的真实数据）：

> 江苏业务组 55 个客户，我拉了一下当前状态：
> - 20 个客户近 15 天日均运行时长 = 0（36% 休眠）
> - 19 个客户健康度 = 0，其中 17 个还标着"合作中"
> - 23 个客户被系统预估流失（42%）
> - 37 个客户超 60 天没有跟进记录（67%）
> - 高级账号开通率只有 36%，819 个额度里有 522 个躺着没用

**点出痛点**：
- BOSS 系统数据全，但看一个客户要点 5 层菜单，没法全局扫描
- CS 每天打开 BOSS 不知道"今天先找谁"

**看板的目标**（一句话）：
> 把 BOSS 系统里的数据变成"CSM 今天该做什么"的工作台，不是"发生了什么"的状态板。

---

### 环节 2：数据从哪来（3 min）

**一图带过**，不展开技术细节：

```
BOSS 系统
  ├─ 客户主数据（55个客户）        → latest-clients.json
  ├─ 合同数据                      → latest-contracts.json
  ├─ 到期合同分析                  → contracts-expiration-summary.json
  └─ 每日使用日报（2年/21000+条）  → latest-reports.json
                    ↓
          build_data.py 合并注入
                    ↓
          dashboard.html（单文件）
                    ↓
          飞书发送 → 浏览器打开即用
```

**关键卖点**（对非技术同事）：
- 一键刷新：`python3 refresh_dashboard.py`，当天数据已拉过会自动跳过
- 单 HTML 文件，17MB，无依赖，飞书传文件直接发，双击打开就是完整看板
- 不需要部署、不需要账号、不需要网络（数据已内嵌）

### 环节 3：看板核心功能演示（18 min）

**建议用真实数据现场演示**，按以下路径走：

#### 3.1 默认入口：指标观察 · 续费季度视图（3 min）

打开看板默认看到的是：**指标观察模式 → 续费季度分组 → 运行次数 → 环比**。

演示要点：
- 默认按续费季度分组（Q1/Q2/Q3/Q4/不在续费期），因为续费是 CS 工作的主节奏
- 每个客户卡片：当前指标值 + 环比变化 + 活跃应用 + 趋势图
- 趋势图里有两条线：实线是当期，虚线是上一期（环比基线）
- 切换指标（演示切到"运行时长"和"高级账号使用率"）
- 切换排序方向（从低到高 / 从高到低）
- 切换到"指标排序"视图：所有客户按指标值平铺排序

**话术**：
> "我想看这个季度要续费的客户里，谁的使用数据最差——切到续费季度视图，按运行次数从低到高排，第一个就是要优先关注的。"

#### 3.2 CSM 诊断模式（5 min）

切到 CSM 诊断模式，演示：

- **Top 10 今日优先跟进**：综合风险、客户价值、续费窗口、跟进超期排序，expired/data-check 不进 Top 10
  - 点开一个 rescue 客户，hover 诊断标签看"命中依据"
  - 点开详情抽屉

- **摘要卡片可点击聚焦**：点"续费抢救"卡片，下方只显示该队列

- **行动队列**（12 个，按优先级排列）：
  - 续费抢救 / 已过期待挽回 / 临期健康续费 / 新签未激活
  - 使用骤降 / 健康度低 / 大客户波动 / 超期未跟进
  - 机会客户 / 个人版转化追踪 / 稳定运营 / 数据异常待核查
  - 每个队列有定义 + 建议动作

**话术**：
> "早上打开看板，先看 Top 10 把今天必跟的过一遍，再按队列扫一遍自己管辖的客户。诊断标签 hover 上去能看到为什么被分到这个队列——不是黑盒。"

#### 3.3 客户详情抽屉（4 min）

点任意客户卡片打开右侧抽屉，演示：

- **9 个指标图同时展示**（运行次数/运行时长/活跃应用/新建应用/高级账号活跃/账号饱和度/健康度/高级账号使用率/有效运行占比）
- 切换列数（1/2/3 列）
- 拖拽调整抽屉宽度
- 独立对比口径（环比/同比/不对比），不影响外部看板
- 独立日期范围控制
- 合同明细区
- 底部"在 BOSS 系统中查看完整信息"跳转按钮

**话术**：
> "跟进一个客户时，点开详情，9 个指标一眼看全，不用在 BOSS 里逐个翻。需要更深的信息再一键跳 BOSS。"

#### 3.4 筛选与配置（3 min）

- 点"筛选"按钮展开筛选面板：服务阶段 / 客户分层 / 续费季度 / CS 负责人
  - 演示只选"本年度新签" → 看新签客户的激活情况
  - 说明默认隐藏"已流失"
- 点"卡片字段 ⚙"：自定义卡片显示哪些字段，选择存 localStorage
- 演示"统一 Y 轴量纲"开关
- 演示底部"指标定义与对比口径说明"折叠区

#### 3.5 持久化（1 min）

- 刷新页面，时间段/指标/对比/筛选/视图/排序/卡片字段/抽屉宽度/抽屉列数全部保留
- 关键点：这些存在浏览器 localStorage，换电脑需要重新设置

### 环节 4：设计亮点（8 min）

挑 5 个值得展开讲的设计决策，每个 1-2 分钟：

#### 亮点 1：双模式分离"客观"与"主观"

| 指标观察模式 | CSM 诊断模式 |
|------------|------------|
| 纯客观数据 | 系统判断 + 行动建议 |
| 不显示预估续费类型、合同金额 | 预估续费类型用颜色区分 |
| 按 续费季度 / 指标排序 组织 | 按 行动队列 / 优先级 组织 |
| Y 轴统一量纲（默认开） | Y 轴各自适应（关注单客户趋势） |

**为什么重要**：数据观察和行动决策是两件事，混在一起会让 CS 分不清"这是事实"还是"这是系统建议"。

#### 亮点 2：按服务阶段分口径判断

同一个"低使用"现象，在不同阶段的含义完全不同：
- 本年度新签 + 低使用 → onboarding 风险 → 安排启用辅导
- 首年续费 + 下滑 → 续费风险 → 紧急续费沟通
- 多年续费 + 低使用 → 场景萎缩 → 探讨扩容

看板的 12 个行动队列不是按指标阈值硬切，而是先看服务阶段再叠合使用数据。

#### 亮点 3：对比描线叠加

选了环比/同比后，图表里不止一个数字，而是两条线：
- 当期实线 + 对比期虚线
- 环比：左历史 + 右当期，时间不重叠
- 同比：同一时间轴重叠，今年 vs 去年

**为什么重要**：一个 "+65.5%" 没有基线参考毫无意义。CS 需要看到"对比的是什么"。

#### 亮点 4：商业价值权重

Top 10 排序不是纯按使用下滑程度，而是叠加了：
- 客户分层（SKA 45 分 / KA 30 分 / SMB 15 分）
- 续费窗口紧迫度（越近越优先）
- 跟进超期程度
- 使用下滑幅度

一个 SKA 客户下降 20%，可能比一个 SMB 下降 80% 更该先跟。

#### 亮点 5：单文件分发

- 17MB 单 HTML，数据全内嵌
- 飞书发文件 → 双击打开 → 完整看板，无需部署
- CSM 不需要技术背景，不需要装环境
- 这个设计决策是有代价的（性能、文件大），但换来了"零门槛分发"

### 环节 5：怎么用起来（3 min）

#### 获取看板

```
我每周一/周三刷新一次数据，通过飞书发给各位
打开 dashboard.html 即可使用
```

#### 日常使用节奏建议

| 频率 | 动作 | 模式 |
|------|------|------|
| 每天 10 分钟 | 看 Top 10，把今天必跟的过一遍 | CSM 诊断 |
| 每周 30 分钟 | 按行动队列扫一遍自己管辖的客户 | CSM 诊断 |
| 每月 1 小时 | 切指标观察，按续费季度看下个季度的续费客户数据 | 指标观察 |
| 跟进单个客户 | 点开详情抽屉，9 指标 + 合同 + 跳 BOSS | 任一模式 |

#### 注意事项

- 数据不是实时的，是最近一次刷新的快照（通常 1-3 天前）
- 看板里看不到跟进记录的详细内容，只有"最近跟进时间"
- 合同金额来自到期合同摘要，不是全量合同
- 需要看最全最新的数据，点详情抽屉里的 BOSS 跳转链接

### 环节 6：后续规划 + 讨论（5 min）

#### 已规划但未做

| 方向 | 状态 |
|------|------|
| 续费季度进度统计（已确认/跟进中/未启动） | 需要本地标记交互，后续迭代 |
| stable 队列二级标签（健康稳定 vs 低活跃稳定） | 规则待沉淀 |
| 浅色主题 | 内部工具优先级低 |
| 跟进记录时间线 | 数据源未覆盖 |
| 行动建议具体执行步骤 | 业务规则待沉淀 |

#### 讨论话题

1. **行动队列的规则准不准**：12 个队列的判定逻辑是否贴合实际？有没有该分没分的、不该分却分了的？
2. **Top 10 排序合不合理**：优先级权重是否需要调整？比如跟进超期是不是应该权重更高？
3. **还缺什么数据**：当前看板没有但你觉得 CS 日常需要的？
4. **分发频率**：每周一次够不够？需要每天吗？
5. **谁想试着自己刷新**：refresh_dashboard.py 一行命令，可以教感兴趣的人自己跑

## 三、演示准备清单

### 演示前 30 分钟

- [ ] 确认 dashboard.html 是最新数据生成的（运行 `refresh_dashboard.py`）
- [ ] 浏览器打开看板，确认无 console 报错
- [ ] 预设好演示路径：先走指标观察 → 续费季度 → 切 CSM 诊断 → 点 Top10 → 开详情抽屉 → 展开筛选
- [ ] 准备一个具体的客户案例（比如某个 rescue 客户），从头到尾走一遍"看到 → 判断 → 行动"的完整流程
- [ ] 关闭浏览器其他标签页，避免演示时找不到

### 演示时的话术原则

- **先说结论再演示**：每个功能先一句话说"解决什么问题"，再操作
- **用真实客户名**：不要用"某客户"，用团队都认识的客户名更有代入感
- **少切模式**：不要频繁切换，一个模式讲完再切下一个
- **留时间看图**：图表出来后停 3 秒让大家看，不要立刻往下点
